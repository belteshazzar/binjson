PROJECT OVERVIEW
----------------
Prototyped and benchmarked a pointer-based B+ tree implementation where values
are stored separately and referenced by pointers in tree nodes, rather than
storing value objects directly in nodes.

MOTIVATION
----------
The original B+ tree stores value objects directly in leaf nodes. During tree
operations (splits, merges), this requires copying potentially large value
objects, which impacts performance with complex/large data structures.

SOLUTION APPROACH
-----------------
Created BPlusTreePtr class that:
1. Stores each value separately in the file
2. Saves only 8-byte Pointer objects in tree nodes
3. Loads values on-demand during search operations
4. Copies only pointers (not values) during node splits

PERFORMANCE RESULTS
-------------------

Large Values (~3KB complex objects):
  Sequential Inserts:  55% faster ✓
  Random Inserts:      59% faster ✓
  Mixed Operations:    48% faster ✓
  Search Heavy:        60% faster ✓
  Range Queries:       43% faster ✓
  File Size:           61% smaller ✓

Small Values (~10 bytes strings):
  Sequential Inserts:  8% faster ✓
  Random Inserts:      8% faster ✓
  Search Heavy:        17% slower (extra indirection)
  Range Queries:       23% slower (extra indirection)
  File Size:           4% larger (pointer overhead)

KEY FINDINGS
------------
1. Pointer approach dramatically outperforms with large values (50-60% faster)
2. Original approach better for small values in read-heavy scenarios
3. Pointer approach uses 82% less memory (lazy loading)
4. Pointer approach creates 60% smaller files with large values
5. Both implementations have identical APIs (drop-in replacement)

RECOMMENDATIONS
---------------

Use BPlusTreePtr (pointer-based) when:
  ✓ Value objects are large (>1KB)
  ✓ Storing documents or complex nested objects
  ✓ Insert/update operations are frequent
  ✓ Memory constraints exist
  ✓ Storage efficiency is important

Use BPlusTree (original) when:
  ✓ Values are small primitives (<100 bytes)
  ✓ Search/query performance is critical
  ✓ Workload is read-heavy
  ✓ Want simpler, battle-tested code
